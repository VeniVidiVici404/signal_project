Alert Generation System:
The Alert Generation System was designed with a strict emphasis on modularity and the separation of concerns, ensuring the architecture remains highly maintainable and easily extensible for future medical requirements. At the core of this subsystem is the AlertGenerator interface, which acts as the foundational blueprint for evaluating patient data. By utilizing a strategy-based approach, the evaluation logic is decoupled from the system's core. This allows developers to create specific, pluggable implementations—such as the BloodPressureAlertGenerator—without altering the underlying framework. If the hospital later requires a new monitoring metric, such as an ECG tracker, a new generator can simply implement this interface without risking disruptions to existing, stable code.

When a concrete generator detects that a patient's vitals have crossed a critical threshold, it instantiates an Alert object. The Alert class is intentionally designed as a lightweight, immutable data structure. Its sole responsibility is to securely transport the context of the event (the patient ID, the specific medical condition, and the precise timestamp) without managing any complex business logic itself.

Finally, to prevent the individual generators from becoming tangled in complex notification routing, the subsystem employs an AlertManager class, implemented via the Singleton Pattern. Routing critical medical alerts requires a centralized, globally accessible dispatcher to ensure no life-saving alerts are dropped, duplicated, or misrouted, making a Singleton the ideal architectural choice. Generators simply pass their instantiated Alert objects to AlertManager.getInstance().dispatchAlert(), completely abstracting away the communication layers. This ensures strict adherence to the Single Responsibility Principle: the generators evaluate, the alert object transports data, and the manager handles the dispatching.


Data Storage System:
The Data Storage System is engineered to be a centralized, secure, and highly efficient repository for continuous streams of cardiovascular data. At the foundation is the PatientData class, an immutable object that encapsulates a single medical reading, complete with patient identification, the specific measurement type, the exact value, and a precise timestamp.

To manage the massive influx of real-time data, the main DataStorage class is implemented using the Singleton Pattern. This guarantees that all incoming data is routed to a single, unified storage instance in memory, preventing fragmented data pools across the application. The DataStorage class also takes responsibility for data lifecycle management by including a purgeOldData() method, fulfilling the requirement to enforce deletion policies for outdated records to free up memory.

Crucially, the system separates the responsibility of storing data from retrieving data to enforce security and privacy constraints. The DataRetriever class acts as a secure gateway for medical staff. Before querying the DataStorage singleton for historical trends, the DataRetriever utilizes a private verifyAccess() method. This ensures that only authorized personnel with the appropriate access level can pull sensitive medical histories, addressing critical non-functional requirements regarding patient privacy and data governance.


Patient Identification System:
The Patient Identification System bridges the gap between the anonymous, numerical IDs generated by the simulator and the highly sensitive, real-world hospital records . The core entity is the HospitalPatient class, which securely holds the patient's real-world identity, medical history, and their corresponding anonymous simulatorId.

The PatientIdentifier class acts as the primary lookup service. It maintains a secure mapping database and provides the resolveSimulatorId() method to translate the incoming stream of anonymous integers into actionable hospital records. This strict separation ensures that the raw data stream never directly handles or exposes sensitive patient names until it is absolutely necessary for medical intervention .

To maintain data integrity and handle edge cases, the IdentityManager oversees the entire identification process. In a live hospital environment, data streams might occasionally include unregistered or corrupted IDs. Instead of allowing the system to crash or misattribute medical data, the IdentityManager includes a handleMismatchError() method. This allows the system to flag unknown simulator IDs, quarantine the unidentifiable data, and alert administrative staff to register the new patient, ensuring the monitoring system remains robust and fault-tolerant.

Data Access Layer:
The Data Access Layer is designed around the principles of polymorphism and the Adapter Pattern to ensure the core Cardiovascular Health Monitoring System remains completely agnostic to how external data arrives . The architecture revolves around the abstract DataListener interface, which establishes a universal contract for starting and stopping data feeds.

By utilizing this interface, we can create highly specialized subclasses—TCPDataListener, WebSocketDataListener, and FileDataListener—that handle the intricate network protocols and file I/O operations independently. If the hospital later decides to integrate a new communication protocol, developers only need to create a new class implementing DataListener, requiring zero changes to the rest of the application.

Once raw data is received by a listener, it is passed through the DataParser. Because incoming formats can vary wildly (e.g., JSON strings vs. Comma-Separated Values), the parser standardizes the input and validates its integrity before it penetrates deeper into the system. Finally, the DataSourceAdapter orchestrates this layer. It utilizes a given listener to fetch the raw data, passes it to the parser for standardization, and ultimately bridges the gap by handing the clean, uniform data off to the internal Storage and Alert systems.
